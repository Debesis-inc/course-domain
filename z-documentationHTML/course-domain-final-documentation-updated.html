1. Application Overview

course-domain is the course-management microservice for Debesis LMS. Its current implemented responsibility is creating a course document, validating the request, enforcing uniqueness rules, generating a URL-safe slug, saving the document into MongoDB, and returning a structured course response.

1

Implemented REST endpoint

The repository exposes one discovered API: POST /api/v1/courses.

3

code, instructorId, and title are checked case-insensitively before save.

Mongo

Persistence model

Course is a MongoDB document stored in collection course.

CI/CD

Deployment path

GitHub Actions builds and pushes Docker images, then deploys to a DigitalOcean VPS via SSH.

Where it fits: In the wider Debesis LMS architecture, this service owns course creation and course metadata. Other domains such as Auth, User, Enrollment, Assignment, Submission, Notification, and Analytics can consume or reference course records through this service rather than writing course data directly.

2. System Architecture

The implemented backend follows a clean Spring layering style: HTTP request enters a controller, business rules run in the service, persistence is delegated to a Spring Data repository, and MongoDB stores the final document.

import com.course_domain.course_domain.dto.request.CourseRequestDTO;
import com.course_domain.course_domain.dto.response.CourseResponseDTO;
import com.course_domain.course_domain.service.CourseService;
import jakarta.validation.Valid;
import lombok.AllArgsConstructor;
import org.springframework.http.HttpStatus;
import org.springframework.http.ResponseEntity;
import org.springframework.web.bind.annotation.PostMapping;
import org.springframework.web.bind.annotation.RequestBody;
import org.springframework.web.bind.annotation.RequestMapping;
import org.springframework.web.bind.annotation.RestController;

@RestController
@AllArgsConstructor
@RequestMapping("/api/v1/courses")
public class CourseController {
    private final CourseService courseService;

    @PostMapping
    public ResponseEntity<CourseResponseDTO> createCourse(@Valid @RequestBody CourseRequestDTO courseRequestDTO) {
        CourseResponseDTO createdCourse = courseService.createCourse(courseRequestDTO);
        return ResponseEntity.status(HttpStatus.CREATED).body(createdCourse);
    }
}
Open service source
package com.course_domain.course_domain.service.impl;

import com.course_domain.course_domain.dto.request.CourseRequestDTO;
import com.course_domain.course_domain.dto.response.CourseResponseDTO;
import com.course_domain.course_domain.exception.DuplicateCourseFieldException;
import com.course_domain.course_domain.mapper.CourseConverter;
import com.course_domain.course_domain.mapper.CourseMapper;
import com.course_domain.course_domain.model.Course;
import com.course_domain.course_domain.repository.CourseRepository;
import com.course_domain.course_domain.service.CourseService;
import lombok.AllArgsConstructor;
import lombok.extern.slf4j.Slf4j;
import org.springframework.stereotype.Service;

import java.time.Instant;
import java.util.Locale;

@Service
@AllArgsConstructor
@Slf4j
public class CourseServiceImpl implements CourseService {

    private final CourseRepository courseRepository;
    private final CourseMapper courseMapper;
    private final CourseConverter courseConverter;

    @Override
    public CourseResponseDTO createCourse(CourseRequestDTO courseRequestDTO) {
        validateUniqueCourseFields(courseRequestDTO);

        Course course = courseConverter.toEntity(courseRequestDTO);
        Instant now = Instant.now();
        course.setSlug(toSlug(courseRequestDTO.getTitle()));
        course.setCreatedAt(now);
        course.setUpdatedAt(now);

        Course savedCourse = courseRepository.save(course);
        return courseMapper.toResponseDTO(savedCourse);
    }

    private void validateUniqueCourseFields(CourseRequestDTO courseRequestDTO) {
        if (courseRepository.existsByCodeIgnoreCase(courseRequestDTO.getCode())) {
            log.warn("Course code already exists: {}", courseRequestDTO.getCode());
            throw new DuplicateCourseFieldException("code", courseRequestDTO.getCode());
        }

        if (courseRepository.existsByInstructorIdIgnoreCase(courseRequestDTO.getInstructorId())) {
            log.warn("Course instructorId already exists: {}", courseRequestDTO.getInstructorId());
            throw new DuplicateCourseFieldException("instructorId", courseRequestDTO.getInstructorId());
        }

        if (courseRepository.existsByTitleIgnoreCase(courseRequestDTO.getTitle())) {
            log.warn("Course title already exists: {}", courseRequestDTO.getTitle());
            throw new DuplicateCourseFieldException("title", courseRequestDTO.getTitle());
        }
    }

    private String toSlug(String value) {
        return value.toLowerCase(Locale.ROOT)
                .replaceAll("[^a-z0-9]+", "-")
                .replaceAll("(^-|-$)", "");
    }
}

Important actual finding: This project already includes Testcontainers MongoDB in BaseIntegrationTest. That means integration tests should use a temporary MongoDB container when Docker is available, instead of depending on a manually started MongoDB on 127.0.0.1:27017.
Why people still see Connection refused 127.0.0.1:27017: the test properties file still contains a localhost Mongo URI, and the main application properties also use Mongo connection strings. If Testcontainers does not start, Docker is not running, the dynamic property key is wrong, or a test starts the app outside the testcontainer base class, Spring may try localhost and fail.

WireMock

WireMock is present through WireMockTestConfig and WireMockStubs. It starts a dynamic-port WireMock server and includes a sample /health stub. In the current course creation tests, WireMock is infrastructure-ready but not central to the asserted course API behavior.

8. Docker Documentation

The project includes a multi-stage Dockerfile and a Compose file with two services: MongoDB and the Spring Boot app.

Dockerfile explained

# ── Stage 1: Build ──────────────────────────────────────
# Uses a full Maven + JDK image just to compile
FROM maven:3.9-eclipse-temurin-21-alpine AS builder

WORKDIR /app

# Copy pom.xml first — Docker caches this layer
# so dependencies are only re-downloaded when pom.xml changes
COPY pom.xml .
RUN mvn dependency:go-offline -q

# Copy source and build
COPY src ./src
RUN mvn clean package -DskipTests -q

# ── Stage 2: Runtime ────────────────────────────────────
# Throws away Maven and JDK — only keeps the JAR
# Alpine = tiny Linux (~5MB base), eclipse-temurin = official Java
FROM eclipse-temurin:21-jre-alpine

WORKDIR /app

# Create a non-root user — security best practice
RUN addgroup -S spring && adduser -S spring -G spring
USER spring

# Copy only the JAR from the builder stage
COPY --from=builder /app/target/course-domain-0.0.1-SNAPSHOT.jar app.jar

# Document that the app listens on 8080
EXPOSE 8080

# Start the app
ENTRYPOINT ["java", "-jar", "app.jar"]

  # ── Spring Boot App ───────────────────────────────────
  course-domain:
    build: .                   # builds from your Dockerfile
    image: course-domain:latest
    container_name: course-domain
    restart: always
    ports:
      - "8080:8080"
    environment:
      SPRING_DATA_MONGODB_URI: mongodb://admin:********@mongodb:27017/course?authSource=admin
      # Notice: uses "mongodb" (service name) not "localhost"
      # Docker networking lets containers find each other by service name
    depends_on:
      - mongodb                # waits for MongoDB to start first

volumes:
  mongo_data:                  # named volume — survives container restarts

MongoDB service

Runs mongo:8.0, creates the initial database, persists data to mongo_data, and maps port 127.0.0.1:27017:27017 so only the host machine can reach MongoDB from outside Docker.

course-domain service

Builds the Spring image from the Dockerfile and exposes the app on 8080:8080. It receives MongoDB configuration through SPRING_DATA_MONGODB_URI.

Why hostname is mongodb

Inside a Docker Compose network, containers resolve each other by service name. The app must use mongodb:27017, not localhost:27017, because localhost inside the app container means the app container itself.

Port mapping detail: 127.0.0.1:27017:27017 is useful for local Docker testing because your IDE, MongoDB Compass, or local app can reach the MongoDB container through localhost while still avoiding public exposure. For the VPS production compose file, this mapping was removed because the Spring container talks to MongoDB internally through the Docker service name mongodb.
Real VPS issue fixed: deployment failed with failed to bind host port 127.0.0.1:27017/tcp: address already in use. The VPS already had a host MongoDB service listening on port 27017, so Docker could not bind the same port for the MongoDB container. The solution was to remove the MongoDB ports block from the VPS compose file. MongoDB remains private, and the app still connects through mongodb:27017 inside the Compose network.

The workflow in .github/workflows/deploy.yml runs on pushes and pull requests targeting main. Pushes to main build and push Docker images, then deploy to the VPS through SSH.

Pull requests run CI checks. Pushes to main continue into Docker push and VPS deployment.

name: CI/CD — Docker Build & Deploy

# ── WHEN does this run? ──────────────────────────────────
on:
  push:
    branches: [ main ]       # every push to main → full CI + CD
  pull_request:
    branches: [ main ]       # every PR → CI only (no deploy)

jobs:

  # ══════════════════════════════════════════════════════
  # JOB 1 — CI: compile, test, build image, push to Hub
  # ══════════════════════════════════════════════════════
  build:
    name: Build & Push Image
    runs-on: ubuntu-latest   # GitHub spins up a fresh Ubuntu VM for this

    steps:

      # 1. Download your code onto the GitHub runner
      - name: Checkout code
        uses: actions/checkout@v4

      # 2. Install Java 25 on the runner
      #    cache: maven → caches ~/.m2 so dependencies
      #    aren't re-downloaded on every run (saves ~1 min)
      - name: Set up Java 25
        uses: actions/setup-java@v4
        with:
          java-version: '25'
          distribution: 'temurin'
          cache: maven

      # 3. Run your tests
      #    If any test fails → this step fails → pipeline stops
      #    → deploy job never runs → server is safe
      - name: Run tests
        run: mvn test
        # ↑ change to: mvn test   once you have real tests

      # 4. Only log into Docker Hub on pushes to main
      #    No point logging in for PR checks
      - name: Log in to Docker Hub
        if: github.ref == 'refs/heads/main'
        uses: docker/login-action@v3
        with:
          username: ${{ secrets.DOCKER_USERNAME }}  # from GitHub secrets
          password: ${{ secrets.DOCKER_PASSWORD }}  # your Docker Hub token

      # 5. Build the Docker image using your Dockerfile
      #    and push it to Docker Hub with two tags:
      #    - debesis/course-domain:latest     (always newest)
      #    - debesis/course-domain:a1b2c3...  (this specific commit)
      - name: Build & Push Docker image
        if: github.ref == 'refs/heads/main'
        uses: docker/build-push-action@v5
        with:
          context: .          # use Dockerfile in project root
          push: true
          tags: |
            ${{ secrets.DOCKER_USERNAME }}/course-domain:latest
            ${{ secrets.DOCKER_USERNAME }}/course-domain:${{ github.sha }}

  # ══════════════════════════════════════════════════════
  # JOB 2 — CD: pull new image on VPS, restart container
  # ══════════════════════════════════════════════════════
  deploy:
    name: Deploy to DigitalOcean
    runs-on: ubuntu-latest
    needs: build             # waits for JOB 1 to finish successfully
    if: github.ref == 'refs/heads/main' && github.event_name == 'push'
    # ↑ only deploys on actual pushes to main, not PRs

    steps:

      - name: Deploy via SSH
        uses: appleboy/ssh-action@v1.0.3
        with:
          host: ${{ secrets.VPS_HOST }}      # 165.245.223.137
          username: ${{ secrets.VPS_USER }}  # root
          key: ${{ secrets.VPS_SSH_KEY }}    # your private SSH key
          script: |

            # Go to where docker-compose.yml lives on the VPS
            cd /opt/course-domain

            # Pull the new image from Docker Hub
            # (only downloads changed layers — fast after first run)
            docker compose pull

            # Restart containers with new image
            # -d = detached (runs in background)
            # Only recreates containers whose image changed
            docker compose up -d

            # Delete old/unused images to save disk space
            # Your 10GB droplet will fill up otherwise
            docker image prune -f

            # Print confirmation with the image name now running
            echo "✅ Deployed $(docker ps --format '{{.Image}}' --filter name=course-domain)"

10. VPS / Deployment-Level Explanation

Using the existing course-domain-docs-v3.html as the deployment reference, the VPS is the production host where Docker Compose starts MongoDB and the course-domain container. GitHub Actions does not run the app itself; it remotely tells the VPS to pull and restart the deployed containers.

☕🍃

What happens after deployment

  1. GitHub Actions connects to the VPS through SSH.
  2. The VPS receives shell commands in /opt/course-domain.
  3. Docker Compose pulls the new image or rebuilds depending on compose setup.
  4. docker compose up -d starts/recreates changed containers.
  5. MongoDB runs as course-mongodb; the app runs as course-domain.
  6. The app reaches MongoDB through the Docker service name mongodb.
  7. The app is exposed on port 8080; MongoDB should remain private.

Runtime concepts

Environment variables are injected when the container starts, not when Java code is written. Docker volumes preserve MongoDB data after container restarts. restart: always helps services come back after crashes or VPS reboot. Docker networks let containers talk using service names.

12. Problems Found / Professional Improvements

Security

Remove hardcoded secrets

MongoDB credentials appear in application and Compose configuration. Move them to environment variables or GitHub/VPS secrets and never commit real passwords.

Build risk

Align Java versions

The project is intentionally using Java 25. Keep the POM, local IntelliJ SDK, GitHub Actions setup-java step, Docker builder image, Docker runtime image, and any direct VPS Java runtime aligned to Java 25.

CI quality

Do not skip tests in CI/CD

The workflow uses mvn test, which is good. The Dockerfile uses -DskipTests; that can be acceptable only if CI already passed tests before image build. Avoid skipping tests in the actual CI quality gate.

Testing

Keep Testcontainers as the default

The project already uses Testcontainers. Keep expanding that approach so integration tests are stable locally and in CI without relying on a pre-installed MongoDB.

API maturity

Add Swagger/OpenAPI

Expose API docs at a route such as /swagger-ui or /v3/api-docs for frontend and QA teams.

Reliability

Add health checks

Add Spring Boot Actuator or custom /health endpoint so Docker, CI, and VPS checks can verify app and MongoDB readiness.

Profiles

Separate dev/test/prod profiles

Use application-dev.properties, application-test.properties, and production environment variables to avoid accidental remote DB usage.

Deployment

Add rollback strategy

Keep commit-SHA image tags, record the previously running image, and add a rollback command or blue/green style deployment plan.

Observability

Improve logs/monitoring

Use structured logs, container log rotation, uptime monitoring, and alerts for VPS disk, memory, and MongoDB failures.

Coverage

Expand endpoint coverage

Add read/update/delete/list endpoints as the course domain matures, then test success, validation, duplicate, and persistence scenarios for each.
